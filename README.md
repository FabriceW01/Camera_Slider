# ESP32-C3 Camera Slider Firmware V1

## Overview

This project turns a manual 1 meter camera slider into a single-axis motorized motion-control system using:

- `ESP32-C3 SuperMini`
- `DRV8825` in `STEP/DIR` mode
- two end limit switches
- a browser-based control UI served directly from the ESP32 over a Wi-Fi access point

The current PlatformIO project configuration was inspected from the repository and uses:

- PlatformIO platform: `espressif32`
- board: `esp32-c3-devkitm-1`
- framework: `espidf`
- ESP-IDF package version: `5.5.3`

The firmware is modular, non-blocking, and built around a controller state machine. Networking stays responsive while the slider is moving.

## Feature List

- Wi-Fi access point hosted by the ESP32
- mobile-friendly web UI served directly by the firmware
- JSON API for UI and future external integrations
- state machine with `BOOT`, `IDLE`, `HOMING`, `READY`, `MANUAL_JOG`, `RUNNING_PROGRAM`, `PAUSED`, `ERROR`, and `LIMIT_TRIGGERED`
- manual jog mode with buttons and a touch-based continuous control strip
- A/B point workflow
- loop and ping-pong A/B playback
- multi-waypoint keyframe playback
- deterministic timing-based motion planning for repeatable moves
- trapezoidal or triangular motion profiles with acceleration and deceleration
- configurable homing with backoff and slow re-approach
- active-low software-debounced limit switch monitoring
- runtime-only configuration updates through the API

## Hardware Requirements

- ESP32-C3 SuperMini or equivalent ESP32-C3 module compatible with the project build target
- DRV8825 stepper motor driver
- 1.8 degree stepper motor
- GT2 belt drive
- 20 tooth pulley
- 2 limit switches, one at each end
- suitable power supply for the motor, driver, and ESP32 wiring arrangement

## Wiring

Current firmware pin mapping:

- `STEP_PIN` -> `GPIO_NUM_2`
- `DIR_PIN` -> `GPIO_NUM_1`
- `LIMIT_LEFT_PIN` -> `GPIO_NUM_3`
- `LIMIT_RIGHT_PIN` -> `GPIO_NUM_4`

Important wiring notes:

- The DRV8825 is used in `STEP/DIR` mode.
- No `ENABLE` pin is controlled by the ESP32 in this version.
- The driver is assumed to remain enabled in hardware.
- Both limit switches are wired as active-low inputs.
- The firmware enables pull-up logic for the limit inputs, so the idle state is `HIGH` and the pressed state is `LOW`.

## DRV8825 Usage

This firmware drives the DRV8825 as a generic step/direction driver:

- `DIR` is set before motion begins
- direction changes are only applied while `STEP` is low
- a small direction-settle delay is enforced before stepping
- step pulses are generated by an ESP-IDF GPTimer-based engine
- the step pulse high time is held long enough for DRV8825-style timing margins

The implementation is intentionally structured so a different `STEP/DIR` driver can be substituted later.

## Limit Switch Logic

- Switches are normally `HIGH`
- Pressed switches read `LOW`
- Raw switch hits are treated as safety-critical events
- Debounced switch state is also tracked for status and homing logic
- If the active travel direction reaches its matching end switch, motion is stopped safely
- If the opposite-end switch triggers unexpectedly, the firmware still treats it as a fault and stops

## Mechanics Model

Assumed mechanics for the default configuration:

- motor full steps per revolution: `200`
- microstepping: `1/32`
- effective microsteps per revolution: `200 * 32 = 6400`
- pulley teeth: `20`
- belt pitch: `2.0 mm`
- travel per motor revolution: `20 * 2.0 mm = 40.0 mm`
- nominal steps per mm: `6400 / 40.0 = 160.0 steps/mm`
- slider travel: `1000.0 mm`

These values are exposed in the runtime configuration and can be changed without adding persistence yet.

## Default Motion Tuning

The defaults are intentionally conservative for a 1 meter belt-driven camera slider. They prioritize smoothness, repeatability, and mechanical safety over headline speed.

- `max_speed_steps_per_sec = 4000`
- `acceleration_steps_per_sec2 = 1000`
- `jog_speed_steps_per_sec = 2400`
- `homing_speed_steps_per_sec = 1200`
- `homing_slow_speed_steps_per_sec = 400`
- `homing_backoff_steps = 640`
- `limit_debounce_ms = 15`

Why these defaults are conservative:

- They reduce the chance of belt skip and lost steps during initial bring-up.
- They keep homing more predictable while switch polarity and direction are being validated.
- They are a safer starting point before a camera payload is mounted.

## Homing Behavior

Default home reference:

- the left end switch is treated as the default home reference

Default homing sequence:

1. If the home switch is already pressed, move away from it by the configured backoff distance.
2. Move toward the home side at the configured homing speed.
3. Stop on the home switch.
4. Back off.
5. Re-approach slowly.
6. Set the current position to `0`.
7. Mark the system as homed and establish soft bounds from `0` to the configured slider travel.

Absolute moves and programmed runs require a homed system.

## Runtime Configuration

The firmware keeps configuration in RAM only. Nothing is persisted across reboot in this version.

Important runtime fields include:

- mechanics conversion values
- max speed
- acceleration
- jog speed
- homing speed
- homing slow speed
- homing backoff
- limit debounce
- home reference side
- motion profile
- repeat mode
- direction inversion
- AP SSID/password metadata

AP credentials are returned by the API, but changing AP credentials at runtime is not implemented in this version.

Default AP configuration:

- SSID: `KameraSlider`
- password: `slider-setup`
- channel: `6`
- max stations: `4`
- IP: `192.168.4.1`

- VS Code with PlatformIO extension, or a working PlatformIO Core installation

## Connecting to the ESP32 Access Point

1. Power the ESP32.
2. On the phone, connect to Wi-Fi SSID `KameraSlider`.
3. Enter password `slider-setup`.
4. Open `http://192.168.4.1/` in a browser.

## Basic Usage

1. Power up with the carriage unloaded.
2. Connect to the ESP32 access point.
3. Open the web UI.
4. Verify both limit switches report the expected idle state.
5. Run homing.
6. Test small jog moves.
7. Capture A and B or upload a keyframe program.
8. Run motion only after confirming direction and limits are correct.

## Safety Notes

Read this before mounting a camera:

- Verify motor direction before any full-travel move.
- Verify both limit switches behave correctly in the dashboard before homing.
- Verify homing direction and stop behavior with the carriage unloaded.
- Verify DRV8825 motor current, cooling, and supply wiring independently.
- Verify the configured travel matches the real mechanical travel.
- Verify the carriage cannot crash into hard stops before the switches trigger.
- Keep an external emergency power-disconnect method available during testing.
- Do not trust the nominal `160 steps/mm` value until it has been measured on the real hardware.

## Assumptions

- DRV8825 is wired in `STEP/DIR` mode.
- The motor is `1.8 degree`, therefore `200` full steps per revolution.
- Microstepping is set to `1/32`, therefore `6400` microsteps per revolution.
- A GT2 `20T` pulley gives `40.0 mm` travel per revolution.
- Nominal conversion is `160.0 steps/mm`.
- No driver `ENABLE` pin is controlled by the ESP32.
- This version is single-axis only.
- No persistence is implemented in this version.
- Real-world calibration is still required.

## Tuning Notes

Tune in this order on real hardware:

1. `invert_direction_output` and home side behavior
2. limit switch polarity and placement
3. `homing_speed_steps_per_sec`
4. `jog_speed_steps_per_sec`
5. `max_speed_steps_per_sec`
6. `acceleration_steps_per_sec2`
7. `slider_travel_mm`
8. `steps_per_mm`

If the slider skips steps, vibrates, stalls, or overshoots, reduce speed and acceleration first.
